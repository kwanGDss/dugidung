# 두기둥 v0.3 — 관계의 달력 (6 타임카드)

- 생성일: 2026-04-15
- 상태: 승인 대기
- 선행: v0.2.0 (`docs/superpowers/specs/2026-04-14-dugidung-v0.2-enrichment.md` + 배포됨)

## 한 줄 요약

결과 페이지에 "관계의 달력" 섹션을 추가한다. 6개의 타임카드 — 지금 / 1~3개월 / 3~6개월 / 6~12개월 / 1년 / 3년 이후 — 각 카드는 그 시점의 분위기 예측 + 그때 할 구체 행동을 담는다. LLM 한 번의 호출로 편지·아키타입·계절카드·달력 모두 같이 생성한다. Letter 타입에 `timeline?: TimePoint[]` 를 optional 로 추가해 기존 v2 캐시와 호환성 유지.

## 목적

- v0.2 까지 "지금" 의 분석은 충분하지만 "앞으로" 방향이 비어있음
- 친구에게 공유했을 때 "우리 1년 뒤 이렇대 ㅋㅋ" 같은 새 훅 추가
- 사주의 전통적인 "운의 흐름" 을 현대적으로 재해석

## 비목표

- 실제 예언·점술 (생일별 유명 점성술 캘린더 매핑 X — LLM 의 서술적 예측만)
- 월별 12 카드 (6개로 제한, 정보 과부하 방지)
- 사용자가 날짜를 선택해 임의 시점 조회 (6 시점 고정)

## 1. 데이터 모델

### 1-1. `lib/types.ts` 확장

```ts
export interface TimePoint {
  when: string;    // 시점 라벨, 2~30자 — "지금 · 0~1개월" 형태
  mood: string;    // 그 시점의 분위기 예측, 30~260자 (1~2문장)
  action: string;  // 그 시점의 구체 행동, 20~200자 (1문장)
}

export interface Letter {
  title: string;
  body: string;
  pullQuote: string;
  archetype: Archetype;
  seasons: Seasons;
  timeline?: TimePoint[];       // NEW — 6 items when present
  promptVersion: string;
  source: "llm" | "fallback";
}
```

### 1-2. `CompatRecord.version` 유지

**`version: 2` 그대로**. 스키마 확장이 additive 이고 `timeline` 은 optional 이므로 기존 v2 레코드와 완전 호환. 결과 페이지는 `timeline` 존재 여부를 null-check 해서 조건부 렌더.

### 1-3. 시점 6개 고정 라벨

프롬프트와 폴백 편지 양쪽에서 동일하게 사용:

| # | label | 의미 |
|---|---|---|
| 1 | `지금 · 0~1개월` | 현재 진행 중인 분위기 |
| 2 | `가까운 미래 · 1~3개월` | 다음 한 계절 |
| 3 | `한 계절 · 3~6개월` | 계절이 바뀌는 시점 |
| 4 | `반년 · 6~12개월` | 중기 |
| 5 | `1년` | 이 관계의 큰 분기점 |
| 6 | `먼 미래 · 3년 이후` | 상상적, 사주의 흐름 기반 추론 |

### 1-4. 불변식

- `timeline` 이 있으면 반드시 정확히 6개 원소
- 각 원소는 `{when, mood, action}` 3 필드 모두 필수
- LLM 생성 실패 시 → 폴백 편지의 timeline 으로 대체 (또는 optional 이므로 아예 생략 가능 — 폴백 정책은 §2-5 참조)

## 2. LLM 프롬프트 변화

### 2-1. `PROMPT_VERSION = "2.1"` (`lib/prompts.ts`)

v2.0 에서 v2.1 로 소폭 bump. `Letter.promptVersion` 에 "2.1" 기록. 기존 v2.0 레코드는 그대로 유지 (timeline 없이).

### 2-2. 단일 호출 유지

편지 + 아키타입 + 계절 3카드 + **타임라인 6카드** 전부 한 번의 호출로. 출력 토큰 ~650 → ~1200 (증가 ~550).

### 2-3. system prompt 추가 블록

기존 규칙 유지 + 아래 블록 append:

```
추가로, "관계의 달력" 6 카드를 생성한다. 각 카드는 한 시점의 분위기 예측 + 그때 할 행동.

시점 6개는 고정 — 순서와 when 라벨 문자열을 정확히 그대로 사용할 것:
1. "지금 · 0~1개월"
2. "가까운 미래 · 1~3개월"
3. "한 계절 · 3~6개월"
4. "반년 · 6~12개월"
5. "1년"
6. "먼 미래 · 3년 이후"

각 카드 필드:
- mood: 30~200자, 1~2문장. 그 시점의 감정·긴장·균형·계절감. 예언조("~할 것입니다") 금지 — 관찰 톤으로.
- action: 20~160자, 1문장. 그 시점에 구체적으로 할 행동. 추상 금지.
  나쁨 예: "서로를 더 사랑하세요"
  좋음 예: "금요일 저녁에 둘이 산책하러 나가기"

톤은 편지와 동일 — 담담, 자연물 비유, 과장 금지.
먼 미래(3년 이후) 는 상상적이어도 좋지만 희망·절망 어느 쪽으로도 과장 금지.
```

### 2-5. zod 스키마 (`lib/llm.ts`)

```ts
const TimePointSchema = z.object({
  when: z.string().min(2).max(40),
  mood: z.string().min(10).max(260),
  action: z.string().min(5).max(200),
});

const LetterSchema = z.object({
  // ... 기존 필드
  timeline: z.array(TimePointSchema).length(6).optional(),
});
```

**`optional` 이유**: LLM 이 가끔 timeline 필드 누락한 JSON 을 반환해도 폴백으로 빠지지 않고 나머지 필드로 성공 처리. 결과 페이지는 timeline 없으면 해당 섹션만 안 보이고 나머지 렌더. 품질 안정성 ↑.

### 2-6. 폴백 편지 확장 (`data/fallback-letters.ts`)

기존 4 버킷(low/mid/high/peak) 각각에 timeline 6개 수동 작성 추가. 폴백도 "완전한 결과" 원칙 유지. 예 (high 버킷):

```ts
high: {
  // ... 기존 필드
  timeline: [
    { when: "지금 · 0~1개월", mood: "속도가 다른 날이 잦다. 대화가 길어지는 것보다 짧게 자주가 낫다.", action: "오늘 밤 자기 전 10분, 얼굴 마주 보고 앉아 있기." },
    { when: "가까운 미래 · 1~3개월", mood: "익숙함이 편안함으로 바뀐다. 작은 습관 하나가 자리잡는 시기.", action: "이번 주말 같이 장을 보고, 새 음식 하나 함께 만들기." },
    { when: "한 계절 · 3~6개월", mood: "열기가 조금 식는 시기. 서로의 빈 자리가 보이기 시작한다.", action: "각자 혼자 있는 시간도 의도적으로 두기. 잠깐 떨어져 있기." },
    { when: "반년 · 6~12개월", mood: "관계의 결이 정해진다. 익숙한 길이 편하지만 지루하기도 하다.", action: "한 번도 안 가본 동네에 같이 가서 반나절 돌아다니기." },
    { when: "1년", mood: "이 관계가 어디로 가는지 한 번 고개 들어 확인하는 시점.", action: "서로의 1년 뒤 모습을 각자 상상해 말로 꺼내 보기." },
    { when: "먼 미래 · 3년 이후", mood: "남아 있다면 불씨보다 장작이 되어 있다. 타오르진 않아도 꺼지지 않는다.", action: "오늘의 작은 선택 하나가 3년 뒤의 자리를 만든다." },
  ],
}
```

low/mid/peak 도 같은 구조로 작성 (톤은 각 버킷의 정서에 맞춤).

## 3. UI 변화

### 3-1. 신규 컴포넌트 — `components/TimelineCards.tsx` (server)

```tsx
import type { TimePoint } from "@/lib/types";

export default function TimelineCards({ timeline }: { timeline: TimePoint[] }) {
  return (
    <div className="my-8">
      <div className="text-[10px] tracking-[0.2em] text-muted uppercase text-center mb-3">
        관계의 달력
      </div>
      <div className="grid grid-cols-1 sm:grid-cols-2 gap-3">
        {timeline.map((t, i) => (
          <div key={i} className="border border-line p-4 rounded-sm">
            <div className="text-[10px] tracking-[0.15em] text-accent uppercase mb-2 flex items-center gap-2">
              <span className="inline-block w-1.5 h-1.5 rounded-full bg-accent" />
              {t.when}
            </div>
            <p className="text-[13px] text-ink leading-relaxed mb-3">
              {t.mood}
            </p>
            <p className="text-[11px] text-muted leading-snug pt-2 border-t border-line">
              <span className="text-accent">→ </span>
              {t.action}
            </p>
          </div>
        ))}
      </div>
    </div>
  );
}
```

### 3-2. 레이아웃

- **모바일 (기본)**: 1열. 각 카드가 화면 폭 꽉 차서 mood/action 읽기 편함. 6개 카드 = 6 행 스크롤.
- **데스크탑 (≥640px, `sm:`)**: 2열. 3 행 그리드, 화면에 2~3개 동시 표시.

### 3-3. 결과 페이지 삽입 위치 (`app/r/[hash]/page.tsx`)

기존 11 섹션 → 12 섹션. 삽입 위치: **"관계의 계절" 과 "요약" 사이**.

최종 순서:
1. 브랜드 행
2. 스코어 링 + 편지 제목
3. 아키타입 배지
4. 네 기둥 카드
5. 오행 분포
6. 차원 바
7. 편지 본문
8. 관계의 계절 (현재 시점 3면)
9. **관계의 달력 (NEW, 6 타임카드)** ← 삽입
10. 요약
11. 공유 바
12. CTA

### 3-4. 조건부 렌더링

```tsx
{rec.letter.timeline && <TimelineCards timeline={rec.letter.timeline} />}
```

기존 v2.0 캐시 레코드는 `timeline` 필드 없음 → 섹션 안 보임. 새로 생성되는 v2.1 레코드부터 자동 렌더.

## 5. 에러 처리 (변경 없음)

- LLM 호출 실패 → 2회 재시도 → 폴백 편지 (이제 timeline 포함)
- LLM 응답 파싱 실패 (zod) → 폴백 편지
- LLM 응답에 timeline 누락 → 그대로 성공 (optional) → 결과 페이지는 해당 섹션 생략
- KV get/set 실패 → v0.2 와 동일
